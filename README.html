Local WordPress Tooling

iota

Spin up isolated WordPress environments with a single command. WordPress, plugins, and themes are managed via Composer. All sites share one MySQL server and one Docker image.

macOS · Linux Apache + PHP 8.2 Composer managed Shared MySQL 8 mkcert SSL

Installation

One-time setup. After this, iota works from any directory.

1

Install dependencies via Homebrew

brew install mkcert
brew install --cask docker
2

Clone this repo anywhere on your Mac

git clone https://github.com/you/iota ~/iota
cd ~/iota
3

Run the installer — symlinks iota to /usr/local/bin and installs the mkcert CA.

chmod +x install.sh
./install.sh
4

The first iota start builds the Docker image automatically. It takes a minute once and is cached after that.

Quick start

# Create a new site
iota create clientsite clientsite.local

# Start it (builds Docker image on first run)
iota start clientsite

# Install WordPress + plugins via Composer
iota compose clientsite
# → generates a template composer.json if none exists

# Open in browser
iota open clientsite
# → https://clientsite.local  (green padlock, no warnings)

Composer workflow

WordPress core, plugins, and themes are all Composer dependencies — nothing is baked into the container. This means updating a plugin is just bumping a version number.

01
Create and start the site: iota create mysite mysite.local then iota start mysite
02
Run iota compose mysite — if no composer.json exists a template is generated automatically with WordPress core and the correct installer paths pre-configured.
03
Edit sites/mysite/composer.json to add plugins from wpackagist.org, e.g. "wpackagist-plugin/woocommerce": "^9.9.3"
04
Run iota compose mysite again to install. To update later, bump the version and re-run.

After composer install your site directory looks like:

sites/mysite/
├── composer.json        ← you own this
├── composer.lock
├── wp-config.php        ← generated by iota create
└── public/              ← Composer output
    ├── wp/              ← WordPress core
    └── app/
        ├── plugins/
        ├── mu-plugins/
        └── themes/

Apache serves public/wp/ as the document root, with /app aliased to public/app/ for plugins and themes.

Commands

Shared services

iota services:startStart shared services — runs automatically via create and start
iota services:stopStop shared services
iota services:statusShow running status of shared services

Site management

iota create <name> [domain]Create a new WordPress site
iota start <name>Start a site's container (builds image if needed)
iota stop <name>Stop a site's container (data preserved)
iota listList all sites with ports and running status
iota destroy <name>Remove container, database, certs, and hosts entry

Composer

iota compose <name>Run composer install inside the container. Generates a template composer.json if none exists.

Database

iota db:import <name> <file>Import a .sql or .sql.gz file
iota db:export <name> [file]Dump the database to a .sql file
iota db:shell <name>Open a MySQL shell scoped to a site's database

Tools

iota wp <name> [args...]Run a WP-CLI command inside the container
iota shell <name>Open a bash shell in the WordPress container
iota logs <name>Tail logs for a site
iota open <name>Open the site in your default browser

File layout

iota/
├── Dockerfile               ← php:8.2-apache + extensions + Composer + WP-CLI
├── services/
│   └── docker-compose.yml   ← shared MySQL + optional S3, Mailpit, Redis
└── sites/
    └── mysite/
        ├── composer.json        ← you own this
        ├── wp-config.php        ← generated by iota create
        ├── docker-compose.yml   ← WordPress container config
        ├── public/              ← Composer output (gitignore)
        │   ├── wp/              ← WordPress core
        │   └── app/             ← plugins, themes, mu-plugins
        ├── certs/               ← mkcert SSL cert + key
        └── apache/              ← Apache vhost config

What persists across restarts

Stopping a site or quitting Docker Desktop loses nothing. Data only disappears with iota destroy.

Survives stop / restart
  • WordPress files (public/)
  • Database (shared MySQL volume)
  • composer.json / composer.lock
  • SSL certificates
  • Local domain in /etc/hosts
Deleted only by destroy
  • Site database & user
  • All WordPress files
  • SSL certs
  • Hosts file entry

Tips

🗄
Database GUI
Connect TablePlus or Sequel Pro once to 127.0.0.1:3306 — every site's database is visible. Root credentials are in services/docker-compose.yml.
📦
Updating plugins
Bump the version in composer.json and run iota compose mysite. Composer handles the rest.
⚙️
Adding services
Uncomment MinIO, Mailpit, or Redis in services/docker-compose.yml and run iota services:start. All containers reach them via the shared network.
🔧
WP-CLI
Run any WP-CLI command: iota wp mysite plugin list, iota wp mysite cache flush, etc.
