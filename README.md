# iota — Local WordPress Development CLI

Spin up isolated WordPress environments with a single command. Each site gets its own Apache container, a trusted SSL cert via Traefik, and a local `.local` domain. All sites share a single MySQL instance and a Traefik reverse proxy. WordPress, plugins, and themes are managed via Composer.

---

## Requirements

- **macOS** (Linux and WSL2 also work)
- **Docker Desktop**
- **mkcert** — `brew install mkcert`

---

## Installation

```bash
git clone <this-repo> iota && cd iota
chmod +x install.sh
./install.sh
```

This symlinks `iota` to `/usr/local/bin/iota` so it works from any directory, and runs `mkcert -install` to set up the local CA so all generated certs are browser-trusted automatically.

The first `iota start` will build the custom Docker image (`iota-php:latest`) from the included `Dockerfile` — this takes a minute once and is cached after that.

---

## Quick Start

```bash
# Create a new site
iota create clientsite clientsite.local

# Start it
iota start clientsite
# → https://clientsite.local  (trusted SSL, no browser warnings)

# Run composer install (generates a template composer.json if none exists)
iota compose clientsite

# Open in browser
iota open clientsite
```

## Composer Workflow

WordPress core, plugins, and themes are all managed via Composer — there is no baked-in WordPress in the container.

```bash
# 1. Create and start the site
iota create mysite mysite.local
iota start mysite

# 2. Drop your composer.json into sites/mysite/
#    (or let iota generate a template)
iota compose mysite

# 3. Edit composer.json to add/update plugins, then re-run
iota compose mysite
```

After `composer install`, your site directory looks like:

```
sites/mysite/
├── .iota                  ← site config (DB credentials, domain, path)
├── composer.json
├── composer.lock
├── wp-config.php          ← generated by iota create (iota-managed)
├── docker-compose.yml     ← generated by iota create (iota-managed)
├── apache/
│   └── iota.conf          ← Apache vhost (iota-managed)
├── certs/                 ← mkcert SSL cert (iota-managed)
└── public/
    ├── wp/                ← WordPress core (Composer output)
    └── app/
        ├── plugins/
        ├── mu-plugins/
        └── themes/
```

Apache serves from `public/wp/` as the document root, with `/app` aliased to `public/app/` for plugins and themes.

### Template composer.json

If no `composer.json` exists when you run `iota compose`, a minimal template is generated:

```json
{
    "name": "iota/site",
    "type": "project",
    "require": {
        "composer/installers": "^2.2",
        "johnpbloch/wordpress": "^6.3.0"
    },
    ...
}
```

Add plugins from [wpackagist.org](https://wpackagist.org) under `require`:

```json
"wpackagist-plugin/woocommerce": "^9.9.3",
"wpackagist-plugin/wordpress-seo": "^25.3",
"wpackagist-theme/hello-elementor": "^3.4.4"
```

Then run `iota compose mysite` again to install them.

## Commands

### Shared Services

| Command | Description |
|---|---|
| `iota services:start` | Start shared services (MySQL, Traefik, Adminer) — runs automatically via `create` and `start` |
| `iota services:stop` | Stop shared services |
| `iota services:status` | Show running status of MySQL, Traefik, and Adminer |

### Site Management

| Command | Description |
|---|---|
| `iota create <name> [domain] [--path <dir>]` | Create a new WordPress site |
| `iota start <name>` | Start a site's container |
| `iota stop <name>` | Stop a site's container (data preserved) |
| `iota list` | List all sites with domain, DB, and status |
| `iota destroy <name>` | Remove site container, database, certs, and hosts entry |

### Composer

| Command | Description |
|---|---|
| `iota compose <name>` | Run `composer install` inside the site container. Generates a template `composer.json` if none exists. |

### Database

| Command | Description |
|---|---|
| `iota db:open` | Open the database tool ([https://db.local](https://db.local)) in your browser |
| `iota db:import <name> <file>` | Import a `.sql` or `.sql.gz` dump |
| `iota db:export <name> [file]` | Export the database to a `.sql` file |
| `iota db:shell <name>` | Open a MySQL shell scoped to a site's DB |

### Tools

| Command | Description |
|---|---|
| `iota info <name>` | Show site config (domain, DB credentials, paths, status) |
| `iota wp <name> [args...]` | Run a WP-CLI command in the container |
| `iota shell <name>` | Open a bash shell in the WordPress container |
| `iota logs <name>` | Tail logs for a site |
| `iota open <name>` | Open the site in your default browser |
| `iota hosts` | Show the hosts file with iota entries highlighted |
| `iota ui` | Interactive TUI (requires [gum](https://github.com/charmbracelet/gum)) |

## Custom Path

Point a site at an existing project directory with `--path`:

```bash
iota create clientsite clientsite.local --path ~/Sites/clientsite
```

All iota resources (certs, apache conf, wp-config, docker-compose) stay in `sites/clientsite/`. The custom directory is mounted as the site content root (`/var/www/html`). If `wp-config.php` already exists in the custom path, it is backed up to `wp-config.php.bak` before iota generates a new one.

`iota destroy` on a custom-path site removes only the iota resources — the project directory is never touched.

## Architecture

### Docker image

The included `Dockerfile` builds `iota-php:latest` from `php:8.2-apache` with:

- All WordPress PHP extensions (`mysqli`, `gd`, `imagick`, `zip`, `intl`, `opcache`, etc.)
- Composer 2
- WP-CLI
- PHP tuned for WordPress (256M uploads, 512M memory, opcache enabled)

Built automatically on first `iota start`. Shared across all sites.

### Shared services

`services/docker-compose.yml` runs three shared services:

- **MySQL** (`iota_mysql`) — shared across all sites, exposed on `127.0.0.1:3306`
- **Traefik** (`iota_traefik`) — reverse proxy, listens on `127.0.0.1:80` and `:443`. Terminates SSL and routes by `Host()` header to the correct WordPress container. HTTP auto-redirects to HTTPS.
- **Adminer** (`iota_adminer`) — database management tool at [https://db.local](https://db.local). Auto-logs in with root credentials. All site databases are visible.

S3/MinIO and Mailpit are stubbed out — uncomment to enable.

### Per-site layout

```
iota/
├── Dockerfile                   ← shared PHP/Apache image
├── services/
│   ├── docker-compose.yml       ← shared MySQL + Traefik + Adminer + optional services
│   ├── adminer/
│   │   └── index.php            ← custom Adminer entry point (auto-login)
│   ├── certs/                   ← Adminer TLS cert for db.local (auto-generated)
│   └── traefik/
│       ├── adminer.yml          ← Adminer TLS config
│       └── <name>.yml           ← per-site TLS cert config (auto-generated)
└── sites/
    └── mysite/
        ├── .iota                ← credentials, domain, content path
        ├── composer.json        ← you own this
        ├── composer.lock
        ├── wp-config.php        ← generated by iota create
        ├── docker-compose.yml   ← WordPress container config
        ├── public/              ← Composer output (gitignore this)
        │   ├── wp/              ← WordPress core
        │   └── app/             ← plugins, themes, mu-plugins
        ├── certs/               ← mkcert SSL cert (used by Traefik)
        └── apache/              ← Apache vhost config
            └── iota.conf
```

### Databases

Each site gets its own database on the shared MySQL:

| Site | Database | User |
|---|---|---|
| `mysite` | `iota_mysite` | `iota_mysite` |
| `clientsite` | `iota_clientsite` | `iota_clientsite` |

`iota create` provisions the database. `iota destroy` drops it.

## What Persists

| | Survives `stop` | Survives `destroy` |
|---|---|---|
| WordPress files (`public/`) | ✓ | ✗ (prompted) |
| Database | ✓ | ✗ |
| `composer.json` / `composer.lock` | ✓ | ✗ (prompted) |
| SSL certs | ✓ | ✗ (prompted) |
| `/etc/hosts` entry | ✓ | ✗ |
| Custom-path project files | ✓ | ✓ (never touched) |

## Adding Services

Uncomment MinIO, Mailpit, or Redis in `services/docker-compose.yml` and run `iota services:start`. All site containers can reach them via the `iota_services` Docker network (e.g. `iota_mailpit:1025` for SMTP).

## Tips

- **Database tool:** `iota db:open` or visit [https://db.local](https://db.local) — auto-logs in and shows all site databases. Use `iota info mysite` to see the DB credentials for a specific site.
- **External database GUI:** connect TablePlus or Sequel Pro to `127.0.0.1:3306` — every site's database is visible. Root credentials are in `services/docker-compose.yml`.
- **Updating a plugin:** bump the version in `composer.json` and run `iota compose mysite` again.
- **WP-CLI:** `iota wp mysite plugin list`, `iota wp mysite cache flush`, etc.
- **First start is slow:** Docker builds the image from scratch. Subsequent starts use the cache and are fast.
- **Interactive mode:** `iota ui` launches a TUI with all commands (requires `brew install gum`).
