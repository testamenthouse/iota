# iota — Claude Code Context

This file gives Claude Code full context on the iota project so it can assist with development, debugging, and extending the system.

---

## What is iota?

`iota` is a local WordPress development CLI for macOS. It lets you spin up multiple isolated WordPress environments as Docker containers, each with:

- A trusted HTTPS local domain (via mkcert + Traefik)
- Apache + PHP 8.2 (custom Docker image)
- WordPress, plugins, and themes managed via Composer
- A dedicated database on a shared MySQL instance

## The CLI (`iota`)

Single bash script (`set -euo pipefail`). All commands are functions prefixed `cmd_`. The router is at the bottom in `main()`.

### Key variables (resolved at runtime)

```bash
SCRIPT_DIR       # absolute path to the iota repo (symlink-resolved)
SITES_DIR        # $SCRIPT_DIR/sites
SERVICES_DIR     # $SCRIPT_DIR/services
MYSQL_ROOT_PASS="iota_root_secret"
MYSQL_CONTAINER="iota_mysql"
TRAEFIK_CONTAINER="iota_traefik"
```

### Key helpers

```bash
get_site_config <name> <KEY>   # reads a key from sites/<name>/.iota; always exits 0
get_site_dir <name>            # returns SITE_PATH from .iota if set, else sites/<name>/
```

### Commands

| Command | Function | Notes |
|---|---|---|
| `iota create <name> [domain] [--path <dir>]` | `cmd_create` | Creates meta_dir, DB, certs, apache conf, wp-config.php, docker-compose.yml, hosts entry |
| `iota start <name>` | `cmd_start` | `docker compose up -d --build` from sites/<name>/ |
| `iota stop <name>` | `cmd_stop` | `docker compose stop` |
| `iota destroy <name>` | `cmd_destroy` | Tears down containers, drops DB, removes hosts entry. For custom-path sites, never touches the project dir. |
| `iota list` | `cmd_list` | Shows all sites, DB names, running status |
| `iota compose <name>` | `cmd_compose` | Runs `composer install` inside container. Generates template composer.json if missing. |
| `iota info <name>` | `cmd_info` | Shows site config: domain, DB credentials, paths, status |
| `iota services:start` | `cmd_services_start` | Starts shared MySQL + Traefik + Adminer (auto-called by create/start) |
| `iota services:stop` | `cmd_services_stop` | Stops shared services |
| `iota services:status` | `cmd_services_status` | Shows MySQL + Traefik + Adminer running status |
| `iota db:import <name> <file>` | `cmd_db_import` | Imports .sql or .sql.gz into site DB |
| `iota db:export <name> [file]` | `cmd_db_export` | Dumps site DB to .sql file |
| `iota db:shell <name>` | `cmd_db_shell` | Opens MySQL shell scoped to site DB |
| `iota shell <name>` | `cmd_shell` | bash shell in WordPress container |
| `iota logs <name>` | `cmd_logs` | Tails all container logs |
| `iota open <name>` | `cmd_open` | Opens https://<domain> in browser |
| `iota wp <name> [args]` | `cmd_wp` | Runs WP-CLI with `--path=/var/www/html/public/wp` |

### Site config file (`.iota`)

Key=value format, one per line:

```
NAME=mysite
DOMAIN=mysite.local
SITE_PATH=/absolute/path/to/content   ← equals sites/<name> for default sites
DB_NAME=iota_mysite
DB_USER=iota_mysite
DB_PASS=<generated>
WP_PREFIX=wp_a1b2c3_
CREATED=2026-04-29T12:00:00Z
```

Read with `get_site_config <name> <KEY>`. Returns empty string (not an error) when a key is absent.

### `--path` / custom-path sites

`iota create mysite mysite.local --path ~/Sites/mysite` separates iota resources from content:

- **`meta_dir`** = `sites/<name>/` — holds `.iota`, `docker-compose.yml`, `certs/`, `apache/iota.conf`, `wp-config.php`
- **`site_dir`** = the custom path — mounted as `/var/www/html` (WordPress content: composer.json, public/, vendor/)

The docker-compose always uses absolute paths for both. `iota destroy` for a custom-path site removes only `meta_dir`; the custom directory is never touched. If a `wp-config.php` exists in the custom path at create time, it is backed up to `wp-config.php.bak`.

## Docker architecture

### Shared image (`iota-php:latest`)

Built from `Dockerfile` in the repo root. Based on `php:8.2-apache`. Includes:

- PHP extensions: `mysqli`, `gd`, `imagick`, `intl`, `bcmath`, `exif`, `opcache`, `zip`
- Composer 2 (copied from `composer:2` image)
- WP-CLI at `/usr/local/bin/wp`
- PHP tuned: 256M uploads, 512M memory, opcache tuned for WordPress

Built automatically on first `iota start` via `--build` flag. Shared across all sites.

### Shared services network

All containers (MySQL, Traefik, per-site WordPress) join the `iota_services` Docker network. This allows WordPress containers to reach MySQL at `iota_mysql:3306` and any future services by container name.

### Traefik reverse proxy

`iota_traefik` is the single entry point for all HTTPS traffic:
- Listens on `127.0.0.1:80` (redirects to HTTPS) and `127.0.0.1:443`
- Routes by `Host()` header to the correct WordPress container
- Terminates SSL using per-site mkcert certs loaded via the file provider
- TLS configs live in `services/traefik/<name>.yml` (mounted as `/etc/traefik/dynamic/`)
- Cert files live in `sites/<name>/certs/` (mounted as `/sites/` inside the container)
- Hot-reloads when new TLS config files appear — no restart needed

### Per-site containers

Each site has one container: `<name>_wordpress`. Volume mounts (always absolute paths):

```yaml
volumes:
  - <site_dir>:/var/www/html                                    # content root
  - <meta_dir>/apache/iota.conf:/etc/apache2/sites-enabled/iota.conf:ro
  - <meta_dir>/wp-config.php:/var/www/html/public/wp-config.php:ro
```

- No published ports — Traefik reaches the container on port 80 via the Docker network
- Serves HTTP only; SSL is terminated by Traefik
- Runs: `a2enmod rewrite headers alias && apache2-foreground`

### MySQL

Single shared `iota_mysql` container. Each site gets its own database (`iota_<name>`) and user (`iota_<name>`). Root password: `iota_root_secret`. Exposed on `127.0.0.1:3306`.

### Adminer

Shared `iota_adminer` container running the official Adminer Docker image. Accessible at `https://db.local` via Traefik. Pre-configured to connect to the shared MySQL instance (`ADMINER_DEFAULT_SERVER=iota_mysql`). Uses the `pepa-linha-dark` design theme.

- TLS cert generated by mkcert at `services/certs/cert.pem` and `key.pem`
- Traefik TLS config at `services/traefik/adminer.yml`
- `db.local` added to `/etc/hosts` during `iota services:start`
- Users can see all site databases (`iota_<name>`) and log in with site credentials from `iota info <name>`

## Composer / WordPress layout

WordPress is NOT baked into the Docker image. It is installed by Composer into the site directory, which is bind-mounted into the container.

### Expected paths (inside container)

| Path | Contents |
|---|---|
| `/var/www/html/public/wp/` | WordPress core |
| `/var/www/html/public/app/plugins/` | Plugins |
| `/var/www/html/public/app/mu-plugins/` | Must-use plugins |
| `/var/www/html/public/app/themes/` | Themes |
| `/var/www/html/public/wp-config.php` | wp-config (bind-mounted from meta_dir) |

### composer.json installer-paths

Must match the above layout:

```json
"extra": {
    "installer-paths": {
        "public/app/plugins/{$name}/":    ["type:wordpress-plugin"],
        "public/app/mu-plugins/{$name}/": ["type:wordpress-muplugin"],
        "public/app/themes/{$name}/":     ["type:wordpress-theme"]
    },
    "wordpress-install-dir": "public/wp"
}
```

**Important:** `wordpress-install-dir` must be the string `"public/wp"`, not the object form. The object form causes installation to the wrong path.

### wp-config.php

Generated by `iota create` into `meta_dir/wp-config.php`, then bind-mounted into the container at `/var/www/html/public/wp-config.php` (one level above ABSPATH). Key constants:

```php
define('WP_CONTENT_DIR', '/var/www/html/public/app');
define('WP_CONTENT_URL', 'https://<domain>/app');
define('WP_HOME',        'https://<domain>');
define('WP_SITEURL',     'https://<domain>');
define('ABSPATH',        '/var/www/html/public/wp/');  // hardcoded — __DIR__ would be wrong
```

ABSPATH is hardcoded because `__DIR__` would resolve to `/var/www/html/public/` (where the file lives), not `/var/www/html/public/wp/` (where WordPress core lives).

## Apache vhost

Single `<VirtualHost *:80>` block per site. SSL is terminated by Traefik — Apache never handles HTTPS.

- Document root: `/var/www/html/public/wp`
- `/app` aliased to `/var/www/html/public/app` (plugins/themes)
- `RequestHeader set X-Forwarded-Proto "https"` so WordPress knows the connection is secure
- WordPress rewrite rules inlined (`AllowOverride None`) — avoids per-request `.htaccess` file reads, significant performance improvement on macOS bind mounts
- `mod_rewrite`, `mod_headers`, and `mod_alias` enabled at container startup

---

## SSL

`mkcert` generates a per-site certificate at `meta_dir/certs/cert.pem` and `key.pem`. `ensure_mkcert()` runs `mkcert -install` every time to guarantee the local CA is trusted by macOS Keychain and browsers.

**How certs flow:** mkcert certs live in `sites/<name>/certs/`. Traefik reads them via the file provider config at `services/traefik/<name>.yml`. Apache never touches SSL.

## Adding a new CLI command

1. Add a `cmd_<name>()` function in `iota`
2. Add a case to the `main()` router at the bottom
3. Add it to the `cmd_help()` printf block

---

## Adding a shared service

1. Uncomment or add a service block in `services/docker-compose.yml`
2. Run `iota services:start`
3. Services are reachable from WordPress containers at `iota_<service>:<port>` via the `iota_services` network

Available stubs in `services/docker-compose.yml`:
- **Mailpit** — SMTP on port 1025, web UI on 8025
- **MinIO** — S3-compatible on port 9000, console on 9001

## Common debugging

### Site shows 403
WordPress files not installed yet. Run `iota compose <name>`.

### 403 with files present
Check Apache document root matches where Composer installed WordPress. Should be `/var/www/html/public/wp`. Inspect with `iota shell <name>` then `ls /var/www/html/public/wp`.

### WordPress shows database setup wizard (wp-config.php not found)
WordPress searches for wp-config.php at ABSPATH (`public/wp/`) then one level up (`public/`). The file is mounted at `public/wp-config.php`. Verify this mount exists in `sites/<name>/docker-compose.yml`. ABSPATH is hardcoded to `/var/www/html/public/wp/` in wp-config.php to compensate for the file living one directory above WordPress.

### Composer installs to wrong path
Check `wordpress-install-dir` in `composer.json` — must be the string `"public/wp"` not an object.

### Container can't reach MySQL
Confirm the `iota_services` network exists: `docker network ls | grep iota_services`. If missing, run `iota services:start`.

### Site not reachable / connection refused on port 443
Traefik must be running. Check with `iota services:status`. If stopped, run `iota services:start`.

### destroy exits silently after site name confirmation
`get_site_config` must always return exit code 0 (it uses `|| true`). If an old `.iota` file is missing a key like `SITE_PATH`, `grep` would exit 1 and with `set -euo pipefail` the script would die. The `|| true` guard prevents this.
